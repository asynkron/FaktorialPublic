How_It_Works

From Backlog To Verified Delivery.
Then To Main.

Each delivery run moves through five stages. Scope, tests, review, CI, and learning stay visible at each step. When judgment is needed, the work pauses and asks for a decision.

search_insights

Investigate

Reads the issue, maps affected files and dependencies, defines the intended scope, and writes acceptance criteria

code

Build

Writes failing tests from acceptance criteria, implements code, runs the suite, and opens a pull request

verified

Verify

Checks CI status, acceptance-criteria coverage, approved scope, test quality, and security scan results

rocket_launch

Deploy

Confirms CI is green, approvals are in, and no conflicts remain. Then merges according to your repo rules

psychology

Learn

Commits decision notes and preventive rules from the run so future issues follow what was learned

Delivery_Engine

Controlled Execution.
Human Approval.

Each issue runs in an isolated workspace, writes tests, implements code, and opens a PR. Multiple issues can move in parallel. When the system is stuck or uncertain, it escalates and waits for your decision.

commit
Issue Isolation

Each issue gets its own git worktree. Parallel work, no file conflicts, no coordination overhead. Real commits, real PRs, clean git history.

target
Scoped Change Protection

The investigate stage declares which files should change. Verify blocks the PR if the diff exceeds that scope. No surprise side effects.

auto_fix_high
Self-Correcting Runs

When tests fail, the system diagnoses the error and fixes it. Runtime failures feed back into the delivery loop instead of becoming noise in your inbox.

Continuous_Learning

Every delivery makes
the next one better.

The learn stage runs after every merge. It captures what happened, why it mattered, and what to do differently next time as repo-committed decision records and preventive rules.

Test Evidence Check

A background check audits closed issues. If an issue has acceptance criteria but no matching test evidence, it raises a follow-up alert so gaps do not quietly disappear.

Decision Records & Preventive Rules

When the learn stage detects retries, friction, or non-trivial decisions, it writes an Architecture Decision Record and a preventive rule. Both are committed to the repo and applied to future runs.

docs/adrs/0096-scope-coverage-per-module.md
# ADR-0096: Scope coverage issues per module
Status: accepted
Issue: #693
## Context
Bundling 3 large modules into one coverage issue
exceeded the 50-turn limit twice before succeeding.
## Decision
Scope each coverage issue to a single module.
.claude/rules/025-scope-coverage-per-module.md
When creating coverage improvement issues,
scope each issue to a single module.
For modules with >1000 statements, expect
2-3x the turns of a typical feature issue.
Reason: Issue #693 bundled 3 modules,
was killed and split. See ADR-0096.
