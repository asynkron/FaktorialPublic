How_It_Works

Your Issues. Our Agents.
Merged to Main.

Every issue flows through five stages. Each stage has quality gates, scope enforcement, and escalation rules. You stay in control. Agents ask for help when they need it.

search_insights

Investigate

Reads the issue, maps affected files, traces dependencies, declares blast radius, and writes an investigation plan with acceptance criteria

code

Build

Writes failing tests from the acceptance criteria (TDD), implements code to make them pass, runs full suite, opens a PR

verified

Verify

Quality gate: checks CI status, AC-to-test traceability, blast radius compliance, test quality level, and security scan results

rocket_launch

Deploy

Validates merge readiness: CI green, approvals obtained, no conflicts. Merges to main and triggers your CI/CD

psychology

Learn

Writes Architecture Decision Records, creates preventive rules from failures, feeds lessons into future agent runs

Agent_Architecture

Controlled Autonomy.
Real Oversight.

Each agent picks up a GitHub issue, checks out its own git worktree, writes tests, implements the code, and opens a PR. Multiple agents work in parallel. When an agent is stuck or uncertain, it escalates and waits for your decision.

commit
Git Worktree Isolation

Each agent gets its own worktree. Parallel work, no file conflicts, no coordination overhead. Real commits, real PRs, clean git history.

target
Blast Radius Protection

The investigate stage declares which files will be touched. Verify blocks the PR if the diff exceeds that scope. No surprise side-effects.

auto_fix_high
Self-Correcting

When tests fail, agents diagnose the error and fix it. Runtime failures feed back into the agent loop — not into your inbox.

Self_Learning

Every delivery makes
the next one better.

The learn stage runs after every merge. It captures what happened, why it mattered, and what to do differently next time, as code-committed ADRs and preventive rules that future agents automatically follow.

Spec-Test Sync Check

The sentinel continuously audits closed issues. If an issue has acceptance criteria but no matching test evidence (PR, merge commit, or child issues delivering the ACs), it raises a follow-up alert, catching gaps that slipped through.

ADRs & Preventive Rules

When the learn stage detects retries, friction, or non-trivial decisions, it writes an Architecture Decision Record and a preventive rule. Both are committed to the repo and applied to all future agent runs.

docs/adrs/0096-scope-coverage-per-module.md
# ADR-0096: Scope coverage issues per module
Status: accepted
Issue: #693
## Context
Bundling 3 large modules into one coverage issue
exceeded the 50-turn limit twice before succeeding.
## Decision
Scope each coverage issue to a single module.
.claude/rules/025-scope-coverage-per-module.md
When creating coverage improvement issues,
scope each issue to a single module.
For modules with >1000 statements, expect
2-3x the turns of a typical feature issue.
Reason: Issue #693 bundled 3 modules,
was killed and split. See ADR-0096.
